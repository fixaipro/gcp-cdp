Security & Privacy Architecture

Privacy-First by Design

Every data product built on a foundation of privacy compliance, consent-first data collection, and enterprise-grade GCP security controls.

🔐
Consent Management

All data collection gated through a Consent Management Platform (CMP). Only explicitly consented signals flow downstream to BigQuery, identity resolution, and ad platform activation.

  • GDPR Article 7 compliant consent capture
  • PECR ePrivacy regulation adherence
  • Consent signals propagate to all downstream DPs
  • Suppression lists for opted-out IDs
  • DSR processing target <72hrs

🏗️
GCP Infrastructure Security

All data within GCP's SOC 2 / ISO 27001 certified environment. Service accounts, IAM roles and VPC Service Controls enforce least-privilege access at every layer.

  • BigQuery column-level access controls
  • Service account-based authentication (no user credentials)
  • VPC Service Controls — data perimeter enforcement
  • Cloud KMS encryption at rest and in transit
  • Cloud Audit Logs for all data access events

🪪
Identity & Data Minimisation

No PII shared with ad platforms in raw form. All identity resolution uses hashed signals. Probabilistic matching used for aggregated insights only — never individual targeting.

  • SHA-256 hashed email + phone for Customer Match
  • Pseudonymised Customer ID (not raw PII)
  • No device IDs or third-party cookies at any layer
  • 2P data via clean room (no raw data share)
  • Data minimisation per DP Privacy Impact Assessment (PIA)

📋
Governance & Compliance

Each data product has an associated PIA and SIA before production deployment. Change management tracked centrally. Data lineage documented in BigQuery Data Catalog.

  • PIA / SIA required for every DP before go-live
  • Data lineage tracked via BigQuery Data Catalog
  • Retention policies enforced via GCP lifecycle rules
  • Third-party data processor agreements in place
  • Annual privacy audit scheduled

🔄
Server-Side Data Flow

Server-side data collection (CAPI / Event Streams) replaces client-side pixel tracking, eliminating browser-based interception risk and ensuring data integrity source to BigQuery.

  • S2S CAPI for Meta & Google signals
  • HTTPS-only endpoints for all data ingestion
  • Server-side tag management (Event Streams)
  • No third-party JavaScript on conversion pages
  • Signed requests for all Connector Hub API calls

🧪
Testing & Validation

Every data product goes through pre-production testing with data quality checks, schema validation, and Test & Learn framework design before activation budget is committed.

  • Pre-prod data quality scoring before promotion
  • Schema validation enforced by Dataform / dbt
  • A/B holdout groups for all audience segment tests
  • Incrementality measurement for all new DPs
  • Analytics team Test & Learn sign-off required

Zero Cookie Dependency: The entire architecture operates without third-party cookies, device IDs, or browser fingerprinting. First-party Customer ID is the single thread connecting all data products — making the platform durable against any future browser or OS privacy changes.

Technical Architecture

Pipeline Stage Breakdown

Each stage of the attribution pipeline — with the actual BigQuery table names, stored procedure logic, and Looker LookML structure.

Stage 1 · Signal Ingestion

Raw Event Collection

All signals land in GCP Cloud Storage as raw JSON or CSV before being processed into BigQuery staging tables by Dataflow pipelines. Server-side collection eliminates browser ITP/Safari blocking risk.

gs://tc-attribution-raw/
  ga4/events/YYYY-MM-DD/*.json
  mmp/postbacks/YYYY-MM-DD/*.json
  meta/capi/YYYY-MM-DD/*.json
  offline/pos/YYYY-MM-DD/*.csv
  • GA4 → BQ native export (events table)
  • MMP S2S postbacks → GCS → Dataflow → BQ
  • Meta/Google CAPI → server-side → GCS staging
  • Offline POS/call → nightly SFTP → GCS → BQ
Stage 2 · Identity Resolution

Unified Profile Assembly

A BigQuery scheduled stored procedure runs nightly to deterministically join all signals to a unified profile ID. Probabilistic fallback fills gaps where deterministic match is unavailable.

-- Deterministic (priority 1)
JOIN ON customer_id = eeid

-- Email hash (priority 2)
JOIN ON SHA256(email)

-- Probabilistic (priority 3)
JOIN ON device_graph_score > 0.8
  • Output: unified_profiles table updated nightly
  • Consent flag enforced — opt-out profiles excluded
  • Match rate monitored via Looker system dashboard
Stage 3 · Touchpoint Assembly

Journey Array Construction

For each converting customer, all touchpoints within the 90-day lookback window are assembled into an ordered journey array. Only consented touchpoints (consent_flag = 1) are included.

SELECT
  unified_profile_id,
  ARRAY_AGG(channel ORDER BY ts) AS journey,
  SUM(revenue) AS total_value
FROM customer_touchpoints
WHERE consent_flag = 1
  AND ts > DATE_SUB(conv_date, 90)
Stage 4 · Shapley Computation

Credit Distribution Logic

The Shapley stored procedure iterates over all 2ⁿ subsets of channels observed in converting journeys. For each subset, it computes the marginal contribution of adding each channel — then averages across all permutations.

CALL shapley_attribution(
  journey_table => 'journey_arrays',
  lookback_days => 90,
  output_table => 'attribution_results_v2',
  include_offline => TRUE,
  channel_cap => 12
);
  • Runs as nightly BigQuery scheduled query (03:00 UTC)
  • Partitioned output by conversion_date for query efficiency
  • Channel cap of 12 prevents combinatorial explosion
Looker Architecture

LookML Design & Dashboard Specifications

📐 LookML Model — Full Structure

+

The attribution LookML model has four Explores joined via unified_profile_id. All metric definitions live in LookML — no per-dashboard SQL. This ensures every team sees the same shapley_credit and marginal_roas numbers regardless of filter state.

-- model: telecom_attribution
connection: "bigquery_production"

explore: attribution_model {
  label: "Cross-Channel Attribution"
  join: channel_spend {
    type: left_outer
    sql_on: ${attribution_model.channel} = ${channel_spend.channel}
    AND ${attribution_model.date} = ${channel_spend.date} ;;
  }
  join: clv_scores {
    type: left_outer
    sql_on: ${attribution_model.unified_profile_id} = ${clv_scores.customer_id} ;;
  }
  join: product_conversion {
    type: left_outer
    sql_on: ${attribution_model.conversion_id} = ${product_conversion.order_id} ;;
  }
}

-- Key measures in attribution_model.view
measure: shapley_credit {
  type: sum
  sql: ${shapley_value} ;;
  value_format: "0.0%"
}
measure: marginal_roas {
  type: number
  sql: ${attributed_revenue} / NULLIF(${channel_spend.spend},0) ;;
  value_format: "#,##0.0x"
}

  SUM(shapley_value) / SUM(spend)

  SUM(attributed_revenue) AS rev

🗺️ Dashboard 2 — Customer Journey Path Explorer

+

A Sankey-style path visualisation showing the most common sequences of channel touchpoints before conversion. Built on the journey_paths_top100 PDT. Dimensions: path length, conversion type, product, CLV tier. Key use: identifies where upper-funnel channels drop off and surfaces the highest-converting sequences for budget reinforcement.

-- journey_paths PDT logic
SELECT
  ARRAY_TO_STRING(journey_array, ' → ') AS path,
  COUNT(*) AS conversion_count,
  AVG(revenue_value) AS avg_order_value,
  ARRAY_LENGTH(journey_array) AS path_length
FROM journey_arrays
WHERE converted = TRUE
  AND consent_flag = 1
GROUP BY 1,4
ORDER BY 2 DESC
LIMIT 100

⚡ Why Shapley Beats Last-Click for Telecoms

+

In telecoms, the average customer journey to contract conversion spans 8–12 touchpoints over 30–60 days. A typical converting journey looks like: TV impression → OOH → Paid Social → Organic Search → Display Retargeting → Brand Paid Search → conversion.

Last-click attributes 100% of credit to Brand Paid Search — the cheapest, lowest-effort touchpoint. This systematically starves upper-funnel investment over time, degrading the pool of demand that lower-funnel can harvest.

EXPECTED BUDGET REALLOCATION AT LAUNCH

When Shapley replaces last-click, typical telecoms outcomes: +15–25% budget shift from brand search/retargeting to prospecting channels (TV, Social, Display). Short-term ROAS appears to drop 10–15% before recovering as the upper funnel rebuilds volume — this should be pre-agreed with Finance before launch.
